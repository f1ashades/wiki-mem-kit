# page_type 涌现与收缩

**这份文档是 kit 层宪法**。它定义两件事：

1. **涌现判断**：sink 时如何评估"这条 raw 是不是该催生一个新 page_type"
2. **收缩判断**：vault 内 type 数量上限和触发条件

各 preset 的 SKILL.md（如 `presets/professional-work/SKILL.md` Step 6）实施"涌现判断"。"收缩判断"是 **future 设计**——`health/SKILL.md` 在 MVP 不实现 type 收缩检查，等 type 数量增长到接近阈值再补；`migrate/SKILL.md` 场景 5 当前依赖人工触发，"自动找合并候选"子流程也是 future。本文档只定义**判断标准**，不写流程。

---

## 设计目标

- **演化**：taxonomy 必须随用户场景成长，不能死在 setup 时定的几个 type 上
- **不爆炸**：避免每次出现新概念都新建 type，导致 50+ type 谁也用不明白
- **不停滞**：避免明明缺类型，但用户不知道，所有信息都塞 meeting body
- **可审计**：每个候选都要写下"为什么提"，让用户事后能 review 决策路径

核心立场：**模型在 sink 时刚读完 raw + 已有 page，是判断"该不该新类型"成本最低的时刻**。但模型的判断必须经过硬性规则过滤，避免泛滥。

---

## 涌现判断：候选生成

每次 sink 末尾隐式跑一遍。命中条件严格分级，不命中不输出，避免噪音。

### Step A：三个触发信号

至少命中一条才进入候选评估。

#### 信号 1：塞不下信号（cramming）

**判断**：本次 sink 是否把某个**有命名的具体实体**强行塞进现有 type 的 body 段或 REVIEW？

**正例**：raw 提到「嘉宝棋牌室」作为一个商家实体，sink 时只能在 meeting 的「会议要点」里写一句"和嘉宝的王五对齐"，没有独立 page 承载这个商家的状态/关系/合作进度。

**反例**：raw 提到"今天天气不错"——不是命名实体，不算塞不下。

#### 信号 2：复发信号（recurrence）

**判断**：同类实体在过去 raw 或 pending-review.md 中出现过 ≥1 次。

**实施**：
- 读 `inbox/pending-review.md` 的「page_type 候选」段，看是否已有同名候选
- 扫描 `raw/` 目录最近 N 个文件（建议 N=10），看相同类别的实体是否反复出现
- 命中即记录"复发次数"

#### 信号 3：显式信号（explicit）

**判断**：用户在 raw 里有明确追踪意图。

**关键词**：
- "我以后要追踪 X"、"这种事会反复发生"、"先记一下，后续可能要单独建档"
- "我们要持续看 X 这个指标"、"X 是我要长期跟的"

显式信号优先级最高——用户明说要追踪的实体，几乎一定该建独立 type 或 page。

### Step B：信号组合阈值

| 命中信号 | 行为 |
|---|---|
| 0 个 | 跳过，不输出候选 |
| 仅 1 个非显式信号 | 跳过 |
| 1 个显式信号 | 进入候选（弱置信度） |
| 2 个非显式信号 | 进入候选（中置信度） |
| 显式 + 任一其他 | 进入候选（强置信度） |
| 全部 3 个 | 进入候选（强置信度，附"立即建议正式化"标签） |

### Step B.5：演化性 gate（**强制 gate，命中即否决**）

**目的**：防止"高频出现但永远不需要独立 page"的实体（如每次开会都提到的"上午那个无名商户"）误触发候选。

判断候选实体是否有**独立演化路径**——即未来是否会有新 raw 写入它：

| 判断 | 处理 |
|---|---|
| **持续追踪对象**：用户对此实体有多次接触意图、关系会演化、信息会增量补充（例：中田这个客户、嘉宝棋牌室、健身助手 SaaS 竞品） | 通过 gate，进入 Step C |
| **一次性事件**：仅此次出现、未来无后续接触意图、信息已经一次性给完（例："上午那个商户"无名无后续、"路过看到的某店"） | **否决**——不写候选，加注 `[否决] {实体名}: 演化性预判为单次事件` |
| **不确定**：模型分不清是哪一类 | 保留候选但置信度强制降到"弱"，并在 pending-review 标注"演化性待观察" |

判断启发式（agent 实操参考）：

- 用户对该实体是否提到了"以后/下次/继续/持续"等未来动作？
- raw 中该实体是否有联系方式、合作意向、未决议题等"延续锚点"？
- 该实体是否有专门的人格化属性（名字、角色、资源），不只是一次提及的标签？
- 满足任一 → 持续追踪；全部不满足 → 一次性事件

**为什么这个 gate 重要**：之前的 Step B 只看"信号数量"，但"塞不下信号"在很多场景下高频出现却低价值——一个临时商户、一次性的提问、一段闲聊都可能算"塞不下"。演化性 gate 直接问"未来会不会回到这条信息"，把"频率高但低演化价值"的候选剔除在前。

### Step C：区分度检查

**目的**：防止"该加字段"的需求被误判成"该新类型"。

对每个候选 type，列出它的**预期字段集合**，与现有每个 type 的字段集合两两比较：

| 字段重合度 | 处理 |
|---|---|
| ≥ 70% | **改写候选**：从"新建 type" 变为"给 X type 加字段 Y"。这是 type-evolution 场景 2（add-optional-field），开销远小于新建 |
| 30-70% | 中间档，候选置信度降一档；提示"可能是边界类，先攒数据再决定" |
| ≤ 30% | 字段正交，候选保留 |

> 区分度判断由模型估算，不需要严格百分比。重要的是触发"该加字段还是该建类型"的反思。

### Step D：体量预估

**目的**：低体量类型是负担——一个 type 长期只有 2-3 个 page，维护成本高于价值。

模型基于 raw 内容和已有上下文，估算这个 candidate type 在 6 个月内大约会产生多少 page：

| 预估 page 数 | 处理 |
|---|---|
| < 5 | **否决**：不写候选，但留一条注记"曾考虑 X type，因体量预估过低未提" |
| 5-20 | **draft 模式**：候选保留，但建议先写到 `inbox/draft-{type}-N.md` 作为草稿，攒满 3 个 draft 再正式建 yaml |
| ≥ 20 | **强候选**：直接建议建 yaml |

> 体量预估是软判断。模型说"约 10"和"约 8"都触发同一档。重点是区分"显然不值得"、"再观察"、"显然该做"三个区间。

### Step E：写入 pending-review.md

最终候选写到 `inbox/pending-review.md` 的固定段：

```markdown
## page_type 候选（来自 sink 自动评估）

- [ ] [置信度: 强] 提议新建 `merchant-profile`
  - 触发信号：塞不下（连锁健身房作为商家实体无独立 page）+ 复发（先前 sink #4 张三 3 家无人棋牌室未建档）
  - 区分度：与 decision 重合 20%、与 meeting 重合 15%（基本正交）
  - 体量预估：6 个月内约 8-15 个 page
  - 建议字段（草案）：merchant_name, business_type, scale, primary_contact, opportunity_summary, mentions_decisions
  - 当前 raw 引用：raw/2026-04-29-健身房教练对话.md
  - 处理建议：建立 yaml（建议 `folder: merchants` 显式覆盖默认 `merchant-profiles/`，目录名更短）；先在 `inbox/draft-merchant-profile-*.md` 攒 2-3 个再正式入该 type 的 resolved folder（按 page-type-spec 的 folder 解析约定）

- [ ] [置信度: 弱] 给 `decision` 加可选字段 `commercial_model`
  - 触发信号：本次 raw 显式提及商业模式细节，与现有「关键约束」段混在一起
  - 区分度：与 decision 重合 100%（即字段加在 decision 上，非新建 type）
  - 体量预估：N/A（字段不预估）
  - 处理建议：参考 type-evolution 场景 2，下一轮 sink 前确认是否加字段
```

**抑制重复**：sink 前先扫描该段已有候选名，已存在的不再追加，但**可以更新**（如复发次数、最新触发 raw）。

### Step F：用户处理

用户 review 候选时可做四件事：
1. **接受**：删除该行，按 type-evolution 场景 1 流程建 yaml
2. **改成加字段**：删除该行，按场景 2 流程加字段
3. **暂缓**：保留该行不动，等下次复发提升置信度
4. **否决**：删除该行，加一行 `- [否决] {type-name}: {理由}`，避免后续 sink 反复提

## 收缩判断：容量管理

随着 vault 规模增长，type 数量必须有上限，否则：
- sink Step 2"判断涉及哪些 page_type"会变慢且嘈杂
- 跨 type 查询成本上升（用户要记 30 个目录）
- 文档维护开销线性增长

### 阈值（建议默认值，可在 config.yaml 覆盖）

> ⏳ **以下全部是 future 行为**。MVP 不实现——`health/SKILL.md` 当前不做 type 数量/密度检查，`migrate/SKILL.md` 当前也不会因 type 多而拒绝 add-type。等 type 数量增长到接近第一行阈值（建议 8-10）时再回来按下表实现。

| 指标 | 阈值 | future 触发动作（MVP 不执行） |
|---|---|---|
| `types_count` | > 25 | **强制收缩**：health 输出 P0；migrate 拒绝继续 add-type，必须先跑合并 review |
| `types_count` | > 15 且月增长 > 1 | **警告**：health 输出 P1，建议月度 review |
| `types_count / total_pages` | > 1/3 | **密度过高**：health 输出 P1（每 type 平均不到 3 页，开销 > 收益） |
| 单 type page 数 | < 5 且存在 > 3 个月 | **低密度类型**：health 列出，建议合并或降级为字段 |

### 收缩执行入口（future）

future 实现后，health/SKILL.md 检查到上述任一阈值时，输出建议但**不自动执行**。

收缩执行走 `migrate/SKILL.md` 场景 5（合并两个 type）的扩展子流程「自动找合并候选」（MVP 未实现，体量到了再加）：

1. 计算两两 type 的相似度（description token 重合 + 字段重合 + 实际 page 文本聚类）
2. 输出 top-3 合并候选 + 各自 page 数 + 合并代价
3. 用户挑哪对合并 → 走标准场景 5 流程

### 不收缩的情况

低 type 数量时（< 8）health 不做收缩检查，因为：
- 数据量小，模型很容易判断 type 边界
- 用户认知负担低，type 多一两个不算问题
- 早期收缩反而会破坏正在涌现的结构

## 设计原则

回看上面所有判断逻辑，背后是同一组立场：

1. **agent 不替用户决定**——agent 提候选，用户决定接受/暂缓/否决
2. **写下来比想清楚重要**——每个候选都要落到 inbox/pending-review.md，让用户隔天能找回上下文
3. **加字段优先于建类型**——区分度检查总是把"加字段"作为第一选择
4. **观察期长于判断期**——5-20 体量的候选要 draft 攒数据，不立刻入 yaml
5. **收缩与涌现对称**——只有涌现没有收缩 = 类型爆炸；只有收缩没有涌现 = 类型停滞
